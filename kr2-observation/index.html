⑤ 结论

1 · 子 agent 对父 system prompt 的命中：结构上为零。公共前缀 0 字符，KV/前缀缓存无法跨父子复用。内容层面仅 10% 重现，且全部是 deepagents 框架自带样板——父的人格设定、workspace 记忆（AGENTS.md/MEMORY.md）、对话历史，子 agent 完全看不到。这是隔离设计的预期行为，但代价是每次委派都是一次冷启动。
2 · 缓存不是"没开"，而是"命中不稳定"——重放实验（cache_probe.py）实锤。把父第 1 轮请求原样重放 3 次：cached_tokens = 0 → 2,624 → 17,024（异步分块提交，64-token 块粒度）；缓存焐热后重放父第 2 轮（前缀扩展）：命中 17,088 / 17,291，证明服务端支持前缀匹配，且暖缓存把时延从 6.5s 降到 2.9s（−55%）。客户端序列化已验证逐字节稳定（两轮工具顺序与内容完全一致），因此原始运行中父第 2 轮的 0 命中是服务端问题：缓存提交延迟 + 网关多副本无会话亲和，命中与否近乎抽签。KR2 行动项：与 tokenhub/GLM 侧确认路由亲和与缓存提交语义；KR1 的 PolicyRouter 应把"缓存亲和度"纳入路由决策并把 cached_tokens 命中率记为 provider 健康指标。
3 · 委派的时间结构：子 episode 占 54%，父被完全阻塞。25.5s 里有 13.9s 在等子 agent 串行跑完（含其内部推理与工具轮次）。KR2 规划中的有界并行调度 + 子 agent 进度流式透出，直接改善的就是这一段。
