# Notes: Windows desktop packaging audit

## Packaging and UX

- Current release builds a Python wheel/sdist and Docker image. There is no PyInstaller, Nuitka, Tauri, Electron, NSIS, Inno Setup, WiX, or MSIX configuration.
- The Windows installer scripts create a uv-managed Python 3.12 virtual environment under `~/.octop`; they do not produce or install a self-contained desktop application.
- The dashboard is a React SPA served by FastAPI from package-relative static files.
- A desktop shell can reuse the SPA without an external browser by embedding Edge WebView2 and running FastAPI on loopback in the same product installation.

## Confirmed Windows support

- GitHub Actions defines a Windows/Python 3.12 job for install, lint, typecheck, and non-live tests.
- Paths use `pathlib` and store mutable data under `Path.home() / '.octop'`.
- Agent workspace root is explicitly adjusted on Windows.
- Native screen capture/input reports Windows as supported when optional desktop dependencies exist.
- Targeted platform tests passed locally: 21 tests.

## Confirmed compatibility gaps

- Windows has no Octop service implementation; only systemd and launchd are detected. The Windows installer nevertheless recommends `octop service start`.
- The interactive terminal explicitly refuses non-POSIX systems.
- SkillHub auto-install invokes `curl` and `bash`; normal Windows installs do not guarantee bash.
- Default agent execution uses deepagents `LocalShellBackend` and `subprocess.run(..., shell=True)`. On Windows this means `cmd.exe`; POSIX command syntax generated by models or bundled skills is not portable.
- The Linux virtual-desktop shell scripts are a server-side desktop-control feature, not a desktop application shell.

## Frozen-runtime gaps

- CLI commands are imported dynamically from strings, so a freezer needs hidden-import collection or a dedicated desktop entry point.
- Dashboard, i18n JSON, SQL migrations, templates, skill libraries, PNGs, and shell assets need explicit data collection.
- Playwright/Chromium is an external runtime asset and should be optional or installed separately.
- Feishu/Yuanbao bot creators spawn `[sys.executable, script.py, ...]`; in a frozen executable `sys.executable` points to the app EXE, so these flows need frozen subcommands or separate helper executables.
- Self-update and plugin dependency installation invoke pip/uv against `sys.executable`; this model is incompatible with a sealed frozen binary.

## Recommended direction

- Windows-only quickest path: Python desktop launcher with pywebview/WebView2, local FastAPI backend, PyInstaller `onedir`, and Inno Setup `Setup.exe`.
- Use a loopback-only random port plus per-launch token. Do not expose the desktop backend on LAN by default.
- Replace system service behavior with startup registration/tray lifecycle.
- Replace in-place pip update with signed installer update.
- Use a PowerShell-aware execution backend on Windows and audit bundled skills for POSIX-only commands.
- Treat browser automation and plugins with native/Python dependencies as optional feature packs for the first desktop release.
